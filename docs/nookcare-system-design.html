Nook and Care – System Design Documentation

Project Overview

This document describes the architecture for Nook and Care, a senior housing marketplace platform designed to help families find verified senior housing facilities.

The goal of the system is to help:

  • Families and Seniors search for and discover verified senior housing facilities
  • Housing Providers list and manage their senior care facilities
  • Platform Administrators verify listings and manage the marketplace

The system allows users to:

  1. Search and discover senior housing listings
  2. View detailed facility listings with images, amenities, and care types
  3. Book tours (in-person or virtual) for facilities
  4. Compare listings side-by-side
  5. Leave reviews and ratings
  6. Manage subscriptions and payments (for providers)
  7. Report fraud and verify providers

The design is built as a marketplace platform supporting many providers and families using the same shared marketplace.

User Stories

Provider (Housing Facility Owner)

As a provider, I want to:

  • Create and manage senior housing listings
  • Upload facility images
  • Configure amenities, care types, and room types for my facilities
  • Set pricing for different room types
  • Manage tour bookings and availability
  • Receive and respond to tour requests
  • View reviews and ratings from families
  • Manage subscriptions (Free, Pro, Premium plans)
  • Upgrade listings to featured status

Family / Senior (Registered User)

As a family member or senior, I want to:

  • Search and filter senior housing by location, care type, budget, and room type
  • View detailed facility information with images and amenities
  • Compare multiple listings side-by-side
  • Book tours (in-person or virtual) for facilities
  • Save favorite listings for later
  • Track my tour bookings
  • Leave reviews and ratings after visits
  • Report fraud or fake listings

Platform Administrator

As an administrator, I want to:

  • Verify providers through business verification workflow
  • Approve or reject listings before publication
  • Review and moderate reports (fraud, fake listings, misleading images)
  • Manage subscription plans and payments
  • Monitor platform analytics and performance
  • Handle verification requests (government license checks, AI image validation)

Industry Workflow

Typical workflow for senior housing marketplace:

  1. Provider Registration: Housing provider creates account and completes business verification
  2. Provider Subscription: Provider subscribes to a plan (Free, Pro, or Premium)
  3. Listing Creation: Provider adds facility details, images, amenities, care types, and room types
  4. Admin Verification: Admin verifies provider credentials and approves listing
  5. Family Search: Family searches for senior housing matching their criteria (location, care type, budget, room type)
  6. Listing Comparison: Family compares multiple listings side-by-side
  7. Tour Booking: Family selects listing, chooses date, and sends tour booking request
  8. Provider Approval: Provider approves tour request
  9. Tour Confirmation: System sends confirmation to family
  10. Property Visit: Family visits facility for tour
  11. Placement: Family decides on placement (outside system scope)
  12. Review Process: Family leaves review and rating after visit
  13. Provider Response: Provider can respond to reviews

System Architecture

The system is designed as a marketplace platform.

Key components:

  • Users (Families/Seniors, Providers, Admin)
  • Senior Housing Listings
  • Tour Bookings
  • Reviews and Ratings
  • Payments and Subscriptions
  • Provider Verification
  • Fraud Reporting

Table Descriptions

1) users

Purpose: Core user accounts for families, seniors, providers, and administrators.

Fields

  • id (UUID, PK)
  • email (VARCHAR(255), UNIQUE)
  • password_hash (TEXT)
  • full_name (VARCHAR(150))
  • phone (VARCHAR(30), NULL)
  • role (VARCHAR(20)) — FAMILY | SENIOR | PROVIDER | ADMIN
  • is_active (BOOLEAN)
  • email_verified_at (TIMESTAMP, NULL)
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • users 1—* providers — A user can become a provider
  • users 1—* tours — Users book tours
  • users 1—* reviews — Users write reviews
  • users 1—* favorites — Users save favorite listings
  • users 1—* verifications — Users can be verified
  • users 1—* reports — Users can file reports

This table is the foundation for authentication and user management.

2) providers

Purpose: Senior housing facility owners who list their care facilities.

Fields

  • id (UUID, PK)
  • user_id (UUID, FK → users.id)
  • business_name (VARCHAR(150))
  • business_type (VARCHAR(50))
  • tax_id (VARCHAR(50), NULL)
  • address (TEXT)
  • city (VARCHAR(100))
  • country (VARCHAR(100))
  • verification_status (VARCHAR(30)) — PENDING | VERIFIED | REJECTED
  • verified_at (TIMESTAMP, NULL)
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • users 1—1 providers — One user account can have one provider profile
  • providers 1—* listings — Providers own multiple facility listings
  • providers 1—* subscriptions — Providers can have subscription plans (Free, Pro, Premium)

This table stores business information for senior housing providers. Providers must complete verification before publishing listings.

3) listings

Purpose: Senior housing facilities available for tours and placement.

Fields

  • id (UUID, PK)
  • provider_id (UUID, FK → providers.id)
  • title (VARCHAR(200))
  • description (TEXT)
  • care_type (VARCHAR(50)) — ASSISTED_LIVING | MEMORY_CARE | INDEPENDENT_LIVING | ADULT_FAMILY_HOME | SKILLED_NURSING
  • capacity (INTEGER) — Total number of residents the facility can accommodate
  • available_beds (INTEGER) — Number of available beds
  • staff_ratio (VARCHAR(50), NULL) — Staff to resident ratio (e.g., “1:3”)
  • established_year (INTEGER, NULL) — Year facility was established
  • phone (VARCHAR(30), NULL) — Contact phone number
  • email (VARCHAR(255), NULL) — Contact email address
  • license_number (VARCHAR(100), NULL) — State license number (if applicable)
  • has_24_hour_care (BOOLEAN) — Whether 24-hour care is available
  • room_type (VARCHAR(50)) — PRIVATE | SEMI_PRIVATE | SHARED
  • address (TEXT)
  • city (VARCHAR(100))
  • state (VARCHAR(100), NULL)
  • country (VARCHAR(100))
  • postal_code (VARCHAR(20), NULL)
  • latitude (DECIMAL(10,8))
  • longitude (DECIMAL(11,8))
  • price (DECIMAL(10,2)) — Monthly or per room pricing
  • currency (VARCHAR(10))
  • is_featured (BOOLEAN) — Featured listings for homepage visibility
  • status (VARCHAR(30)) — ACTIVE | INACTIVE | PENDING | SUSPENDED
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • providers 1—* listings — One provider can have many facilities
  • listings 1—* listing_images — Each listing has multiple images
  • listings 1—* listing_amenities — Listings have amenities
  • listings 1—* listing_services — Listings offer care services
  • listings 1—* listing_equipment — Listings have medical/assistive equipment
  • listings 1—* listing_languages — Listings support languages
  • listings 1—* listing_certifications — Listings have certifications/licenses
  • listings 1—* listing_activities — Listings offer activities and programs
  • listings 1—* listing_dining_options — Listings provide dining options
  • listings 1—* listing_safety_features — Listings have safety features
  • listings 1—* listing_insurance_options — Listings accept insurance/payment types
  • listings 1—* tours — Listings offer tours (in-person or virtual)
  • listings 1—* reviews — Listings receive reviews
  • listings 1—* favorites — Listings can be favorited
  • listings 1—* reports — Listings can be reported for fraud

This is the core entity representing available senior housing facilities. Listings must be verified before publication.

4) listing_images

Purpose: Images associated with senior housing facility listings.

Fields

  • id (UUID, PK)
  • listing_id (UUID, FK → listings.id)
  • image_url (TEXT)
  • display_order (INTEGER)
  • is_primary (BOOLEAN)
  • created_at (TIMESTAMP)

Relationships (why)

  • listings 1—* listing_images — Each listing has multiple images
  • One image should be marked as primary for display purposes
  • Display order allows providers to organize images
  • Images are validated by AI for fraud prevention

5) amenities

Purpose: Master catalog of available amenities (pool, WiFi, parking, etc.).

Fields

  • id (UUID, PK)
  • name (VARCHAR(100))
  • category (VARCHAR(50)) — BASIC | PREMIUM | SAFETY | ACCESSIBILITY
  • icon (VARCHAR(100), NULL)
  • created_at (TIMESTAMP)

Relationships (why)

  • amenities 1—* listing_amenities — Amenities are used across many listings
  • This is a master catalog to ensure consistency

6) listing_amenities

Purpose: Junction table linking listings to their amenities.

Fields

  • id (UUID, PK)
  • listing_id (UUID, FK → listings.id)
  • amenity_id (UUID, FK → amenities.id)
  • created_at (TIMESTAMP)

Relationships (why)

  • listings *—* amenities — Many-to-many relationship
  • A listing can have many amenities, and an amenity can be in many listings

Suggested constraint:

  • Unique (listing_id, amenity_id) to prevent duplicates

Example Tour Booking Flow

Example tour booking process for senior housing:

1. Family searches for senior housing in Los Angeles
2. Filters by: Care Level (Assisted Living), Monthly Budget ($3000-5000), Amenities, Certifications, Insurance Accepted, Availability
3. Views listing details with images, amenities, and care types
4. Compares multiple listings side-by-side
5. Saves preferred listings to favorites
6. Selects a listing and books a tour (in-person)
7. Provider receives tour request and approves
8. System sends confirmation to family
9. Family visits facility for tour
10. After visit, family leaves 4-star review
11. Provider responds to review
12. Family decides on placement (outside system scope)

The system:

  1. Tracks the search and filter criteria (location, care level, monthly budget, amenities, certifications, insurance accepted, availability, ratings)
  2. Stores favorites for later comparison
  3. Manages tour booking workflow (request → approval → confirmation)
  4. Tracks tour bookings and status
  5. Generates reviews after tours
  6. Updates listing statistics and ratings
  7. Supports map-based location search and distance filtering
