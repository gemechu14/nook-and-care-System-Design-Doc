Nook and Care – System Design Documentation

Project Overview

This document describes the architecture for Nook and Care, a senior housing marketplace platform designed to help families find verified senior housing facilities.

The goal of the system is to help:

  • Families and Seniors search for and discover verified senior housing facilities
  • Housing Providers list and manage their senior care facilities
  • Agents access leads and premium data
  • Platform Administrators verify listings and manage the marketplace

The system allows users to:

  1. Search and discover senior housing listings
  2. View detailed facility listings with images, amenities, and care types
  3. Book tours (in-person or virtual) for facilities
  4. Compare listings side-by-side
  5. Leave reviews and ratings
  6. Manage subscriptions and payments (for providers)
  7. Report fraud and verify providers

The design is built as a single-tenant platform supporting many providers and families using the same marketplace.

User Stories

Provider (Housing Facility Owner)

As a provider, I want to:

  • Create and manage senior housing listings
  • Upload facility images
  • Configure amenities, care types, and room types for my facilities
  • Set pricing for different room types
  • Manage tour bookings and availability
  • Receive and respond to tour requests
  • View reviews and ratings from families
  • Manage subscriptions (Free, Pro, Premium plans)
  • Upgrade listings to featured status

Family / Senior (Registered User)

As a family member or senior, I want to:

  • Search and filter senior housing by location, care type, budget, and room type
  • View detailed facility information with images and amenities
  • Compare multiple listings side-by-side
  • Book tours (in-person or virtual) for facilities
  • Save favorite listings for later
  • Track my tour bookings
  • Leave reviews and ratings after visits
  • Report fraud or fake listings

Agent

As an agent, I want to:

  • Access lead dashboard with family inquiries
  • View premium listings and data
  • Contact families and providers
  • Track conversion metrics

Platform Administrator

As an administrator, I want to:

  • Verify providers through business verification workflow
  • Approve or reject listings before publication
  • Review and moderate reports (fraud, fake listings, misleading images)
  • Manage subscription plans and payments
  • Monitor platform analytics and performance
  • Handle verification requests (government license checks, AI image validation)

Industry Workflow

Typical workflow for senior housing marketplace:

  1. Provider Registration: Housing provider creates account and completes business verification
  2. Provider Subscription: Provider subscribes to a plan (Free, Pro, or Premium)
  3. Listing Creation: Provider adds facility details, images, amenities, care types, and room types
  4. Admin Verification: Admin verifies provider credentials and approves listing
  5. Family Search: Family searches for senior housing matching their criteria (location, care type, budget, room type)
  6. Listing Comparison: Family compares multiple listings side-by-side
  7. Tour Booking: Family selects listing, chooses date, and sends tour booking request
  8. Provider Approval: Provider approves tour request
  9. Tour Confirmation: System sends confirmation to family
  10. Property Visit: Family visits facility for tour
  11. Placement: Family decides on placement (outside system scope)
  12. Review Process: Family leaves review and rating after visit
  13. Provider Response: Provider can respond to reviews

System Architecture

The system is designed as a single-tenant marketplace platform.

Key components:

  • Users (Families/Seniors, Providers, Agents, Admin)
  • Senior Housing Listings
  • Tour Bookings
  • Reviews and Ratings
  • Payments and Subscriptions
  • Provider Verification
  • Fraud Reporting

Table Descriptions

1) users

Purpose: Core user accounts for families, seniors, providers, agents, and administrators.

Fields

  • id (UUID, PK)
  • email (VARCHAR(255), UNIQUE)
  • password_hash (TEXT)
  • full_name (VARCHAR(150))
  • phone (VARCHAR(30), NULL)
  • role (VARCHAR(20)) — FAMILY | SENIOR | PROVIDER | AGENT | ADMIN
  • is_active (BOOLEAN)
  • email_verified_at (TIMESTAMP, NULL)
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • users 1—* providers — A user can become a provider
  • users 1—* tours — Users book tours
  • users 1—* reviews — Users write reviews
  • users 1—* favorites — Users save favorite listings
  • users 1—* verifications — Users can be verified
  • users 1—* reports — Users can file reports

This table is the foundation for authentication and user management.

2) providers

Purpose: Senior housing facility owners who list their care facilities.

Fields

  • id (UUID, PK)
  • user_id (UUID, FK → users.id)
  • business_name (VARCHAR(150))
  • business_type (VARCHAR(50))
  • tax_id (VARCHAR(50), NULL)
  • address (TEXT)
  • city (VARCHAR(100))
  • country (VARCHAR(100))
  • verification_status (VARCHAR(30)) — PENDING | VERIFIED | REJECTED
  • verified_at (TIMESTAMP, NULL)
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • users 1—1 providers — One user account can have one provider profile
  • providers 1—* listings — Providers own multiple facility listings
  • providers 1—* subscriptions — Providers can have subscription plans (Free, Pro, Premium)

This table stores business information for senior housing providers. Providers must complete verification before publishing listings.

3) listings

Purpose: Senior housing facilities available for tours and placement.

Fields

  • id (UUID, PK)
  • provider_id (UUID, FK → providers.id)
  • title (VARCHAR(200))
  • description (TEXT)
  • care_type (VARCHAR(50)) — INDEPENDENT_LIVING | ASSISTED_LIVING | MEMORY_CARE | NURSING_HOME | etc.
  • room_type (VARCHAR(50)) — PRIVATE | SEMI_PRIVATE | SHARED
  • address (TEXT)
  • city (VARCHAR(100))
  • state (VARCHAR(100), NULL)
  • country (VARCHAR(100))
  • postal_code (VARCHAR(20), NULL)
  • latitude (DECIMAL(10,8))
  • longitude (DECIMAL(11,8))
  • price (DECIMAL(10,2)) — Monthly or per room pricing
  • currency (VARCHAR(10))
  • is_featured (BOOLEAN) — Featured listings for homepage visibility
  • status (VARCHAR(30)) — ACTIVE | INACTIVE | PENDING | SUSPENDED
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • providers 1—* listings — One provider can have many facilities
  • listings 1—* listing_images — Each listing has multiple images
  • listings 1—* listing_amenities — Listings have amenities
  • listings 1—* listing_services — Listings offer care services
  • listings 1—* listing_equipment — Listings have medical/assistive equipment
  • listings 1—* listing_languages — Listings support languages
  • listings 1—* tours — Listings offer tours (in-person or virtual)
  • listings 1—* reviews — Listings receive reviews
  • listings 1—* favorites — Listings can be favorited
  • listings 1—* reports — Listings can be reported for fraud

This is the core entity representing available senior housing facilities. Listings must be verified before publication.

4) listing_images

Purpose: Images associated with senior housing facility listings.

Fields

  • id (UUID, PK)
  • listing_id (UUID, FK → listings.id)
  • image_url (TEXT)
  • display_order (INTEGER)
  • is_primary (BOOLEAN)
  • created_at (TIMESTAMP)

Relationships (why)

  • listings 1—* listing_images — Each listing has multiple images
  • One image should be marked as primary for display purposes
  • Display order allows providers to organize images
  • Images are validated by AI for fraud prevention

5) amenities

Purpose: Master catalog of available amenities (pool, WiFi, parking, etc.).

Fields

  • id (UUID, PK)
  • name (VARCHAR(100))
  • category (VARCHAR(50)) — BASIC | PREMIUM | SAFETY | ACCESSIBILITY
  • icon (VARCHAR(100), NULL)
  • created_at (TIMESTAMP)

Relationships (why)

  • amenities 1—* listing_amenities — Amenities are used across many listings
  • This is a master catalog to ensure consistency

6) listing_amenities

Purpose: Junction table linking listings to their amenities.

Fields

  • id (UUID, PK)
  • listing_id (UUID, FK → listings.id)
  • amenity_id (UUID, FK → amenities.id)
  • created_at (TIMESTAMP)

Relationships (why)

  • listings *—* amenities — Many-to-many relationship
  • A listing can have many amenities, and an amenity can be in many listings

Suggested constraint:

  • Unique (listing_id, amenity_id) to prevent duplicates

7) services

Purpose: Master catalog of services that can be offered (cleaning, concierge, etc.).

Fields

  • id (UUID, PK)
  • name (VARCHAR(100))
  • description (TEXT, NULL)
  • category (VARCHAR(50))
  • base_price (DECIMAL(10,2), NULL)
  • created_at (TIMESTAMP)

Relationships (why)

  • services 1—* listing_services — Services are offered by multiple listings
  • Base price provides a default, but listing_services can override

8) listing_services

Purpose: Services offered by specific listings with custom pricing.

Fields

  • id (UUID, PK)
  • listing_id (UUID, FK → listings.id)
  • service_id (UUID, FK → services.id)
  • price (DECIMAL(10,2))
  • is_included (BOOLEAN) — Whether service is included in base price
  • created_at (TIMESTAMP)

Relationships (why)

  • listings *—* services — Many-to-many with pricing
  • Allows each listing to set custom prices for services

9) equipment

Purpose: Master catalog of equipment available (kitchen appliances, electronics, etc.).

Fields

  • id (UUID, PK)
  • name (VARCHAR(100))
  • category (VARCHAR(50))
  • description (TEXT, NULL)
  • created_at (TIMESTAMP)

Relationships (why)

  • equipment 1—* listing_equipment — Equipment is used across listings
  • Master catalog ensures consistency

10) listing_equipment

Purpose: Equipment available at specific listings with quantities.

Fields

  • id (UUID, PK)
  • listing_id (UUID, FK → listings.id)
  • equipment_id (UUID, FK → equipment.id)
  • quantity (INTEGER)
  • created_at (TIMESTAMP)

Relationships (why)

  • listings *—* equipment — Many-to-many with quantity tracking
  • Tracks how many of each equipment item is available

11) languages

Purpose: Master catalog of languages supported.

Fields

  • id (UUID, PK)
  • code (VARCHAR(10)) — ISO language code (en, es, fr, etc.)
  • name (VARCHAR(100))
  • created_at (TIMESTAMP)

Relationships (why)

  • languages 1—* listing_languages — Languages are used across listings
  • Standard language codes for consistency

12) listing_languages

Purpose: Languages supported by specific listings.

Fields

  • id (UUID, PK)
  • listing_id (UUID, FK → listings.id)
  • language_id (UUID, FK → languages.id)
  • created_at (TIMESTAMP)

Relationships (why)

  • listings *—* languages — Many-to-many relationship
  • Indicates which languages the provider can communicate in

13) house_rules

Purpose: Rules and policies for specific listings.

Fields

  • id (UUID, PK)
  • listing_id (UUID, FK → listings.id)
  • rule_text (TEXT)
  • display_order (INTEGER)
  • created_at (TIMESTAMP)

Relationships (why)

  • listings 1—* house_rules — Each listing can have multiple rules
  • Display order allows providers to prioritize rules

14) tours

Purpose: Virtual or in-person tours scheduled for senior housing facilities.

Fields

  • id (UUID, PK)
  • listing_id (UUID, FK → listings.id)
  • tour_type (VARCHAR(50)) — VIRTUAL | IN_PERSON
  • scheduled_at (TIMESTAMP)
  • status (VARCHAR(30)) — PENDING | APPROVED | SCHEDULED | COMPLETED | CANCELLED
  • booked_by_user_id (UUID, FK → users.id, NULL)
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • listings 1—* tours — Listings can offer multiple tours
  • users 1—* tours — Users (families/seniors) can book tours
  • Workflow: User selects listing → selects date → sends booking request → provider approves → confirmation sent

15) bookings

Purpose: Tour bookings made by families/seniors for facility tours.

Note: This table may be merged with tours table or kept separate depending on implementation. The SRS mentions tour booking workflow, so this could represent tour bookings rather than accommodation bookings.

Fields

  • id (UUID, PK)
  • listing_id (UUID, FK → listings.id)
  • user_id (UUID, FK → users.id)
  • tour_date (DATE)
  • tour_type (VARCHAR(50)) — VIRTUAL | IN_PERSON
  • status (VARCHAR(30)) — PENDING | APPROVED | CONFIRMED | COMPLETED | CANCELLED
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • listings 1—* bookings — Listings receive multiple tour bookings
  • users 1—* bookings — Users make multiple tour bookings
  • bookings 1—* reviews — Completed tours can generate reviews

This represents tour bookings for senior housing facilities.

16) reviews

Purpose: Reviews and ratings left by families/seniors after facility tours or placement.

Fields

  • id (UUID, PK)
  • booking_id (UUID, FK → bookings.id, NULL) — Optional, if review is after tour
  • listing_id (UUID, FK → listings.id)
  • user_id (UUID, FK → users.id)
  • rating (INTEGER) — 1 to 5
  • comment (TEXT, NULL)
  • response_from_provider (TEXT, NULL)
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • listings 1—* reviews — Listings accumulate reviews
  • users 1—* reviews — Users write multiple reviews
  • Providers can respond to reviews
  • Admin moderates reviews

17) leads

Purpose: Inquiry messages from potential guests to providers.

Fields

  • id (UUID, PK)
  • listing_id (UUID, FK → listings.id)
  • user_id (UUID, FK → users.id)
  • message (TEXT)
  • status (VARCHAR(30)) — NEW | READ | RESPONDED | CLOSED
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • listings 1—* leads — Listings receive inquiry leads
  • users 1—* leads — Users create leads
  • Allows communication before booking

18) favorites

Purpose: Saved listings that users want to revisit.

Fields

  • id (UUID, PK)
  • user_id (UUID, FK → users.id)
  • listing_id (UUID, FK → listings.id)
  • created_at (TIMESTAMP)

Relationships (why)

  • users 1—* favorites — Users can favorite multiple listings
  • listings 1—* favorites — Listings can be favorited by multiple users
  • Many-to-many relationship

Suggested constraint:

  • Unique (user_id, listing_id) to prevent duplicate favorites

19) subscriptions

Purpose: Subscription plans for providers to publish listings. Plans include Free (limited listings), Pro (unlimited listings), and Premium (featured listings).

Fields

  • id (UUID, PK)
  • provider_id (UUID, FK → providers.id)
  • plan_type (VARCHAR(50)) — FREE | PRO | PREMIUM
  • price (DECIMAL(10,2))
  • billing_cycle (VARCHAR(20)) — MONTHLY | YEARLY
  • start_date (DATE)
  • end_date (DATE)
  • status (VARCHAR(30)) — ACTIVE | EXPIRED | CANCELLED
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • providers 1—* subscriptions — Providers can have subscription history
  • Providers must subscribe to publish listings
  • Premium plan enables featured listings with homepage visibility and search ranking boost

20) payments

Purpose: Payment transactions for bookings.

Fields

  • id (UUID, PK)
  • booking_id (UUID, FK → bookings.id)
  • amount (DECIMAL(10,2))
  • currency (VARCHAR(10))
  • payment_method (VARCHAR(50)) — CREDIT_CARD | PAYPAL | BANK_TRANSFER
  • transaction_id (VARCHAR(255), NULL)
  • status (VARCHAR(30)) — PENDING | COMPLETED | FAILED | REFUNDED
  • processed_at (TIMESTAMP, NULL)
  • created_at (TIMESTAMP)

Relationships (why)

  • bookings 1—1 payments — Each booking has one payment
  • Tracks payment processing and status

21) verifications

Purpose: Verification system for providers including government license checks, manual admin review, and AI image validation. Verified providers receive a verification badge.

Fields

  • id (UUID, PK)
  • user_id (UUID, FK → users.id) — Typically for providers
  • verification_type (VARCHAR(50)) — BUSINESS_LICENSE | GOVERNMENT_LICENSE | IDENTITY | IMAGE_VALIDATION
  • document_url (TEXT, NULL)
  • status (VARCHAR(30)) — PENDING | VERIFIED | REJECTED
  • verified_at (TIMESTAMP, NULL)
  • verified_by_admin_id (UUID, FK → users.id, NULL) — Admin who verified
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • users 1—* verifications — Providers can have multiple verification types
  • Ensures platform security and trust
  • Verification includes government license checks, manual admin review, and AI image validation

22) reports

Purpose: Reporting system for users to report fake listings, scam providers, or misleading images. Admin receives alerts.

Fields

  • description (TEXT)
  • status (VARCHAR(30)) — PENDING | REVIEWED | RESOLVED | DISMISSED
  • created_at (TIMESTAMP)
  • updated_at (TIMESTAMP)

Relationships (why)

  • users 1—* reports — Users can file reports
  • listings 1—* reports — Listings can be reported
  • Enables fraud prevention and platform safety
  • Admin receives alerts for review

Example Tour Booking Flow

Example tour booking process for senior housing:

1. Family searches for senior housing in Los Angeles
2. Filters by: Assisted Living, Private Room, Budget $3000-5000/month
3. Views listing details with images, amenities, and care types
4. Compares multiple listings side-by-side
5. Saves preferred listings to favorites
6. Selects a listing and books a tour (in-person)
7. Provider receives tour request and approves
8. System sends confirmation to family
9. Family visits facility for tour
10. After visit, family leaves 4-star review
11. Provider responds to review
12. Family decides on placement (outside system scope)

The system:

  1. Tracks the search and filter criteria (location, care type, budget, room type, ratings, availability)
  2. Stores favorites for later comparison
  3. Manages tour booking workflow (request → approval → confirmation)
  4. Tracks tour bookings and status
  5. Generates reviews after tours
  6. Updates listing statistics and ratings
  7. Supports map-based location search and distance filtering

Key Design Decisions

Why Separate Master Catalogs?

The system uses master catalogs for amenities, services, equipment, and languages to:

  • Ensure consistency across listings
  • Enable filtering by standardized attributes
  • Simplify management of common attributes
  • Support analytics on popular features

Why Junction Tables?

Junction tables (listing_amenities, listing_services, etc.) allow:

  • Many-to-many relationships between listings and attributes
  • Custom pricing for services per listing
  • Quantity tracking for equipment
  • Flexible configuration without denormalization

Why Separate Payments Table?

The payments table is separate from bookings to:

  • Support multiple payment attempts if first fails
  • Track payment history independently
  • Handle refunds as separate transactions
  • Integrate with payment gateways more easily

Why Leads Table?

The leads table enables (if implemented):

  • Pre-tour communication between families and providers
  • Lead tracking for agents to access family inquiries
  • Message history before tour commitment
  • Conversion analytics on inquiry patterns
  • Agent dashboard access to premium leads data

Note: The SRS mentions agents accessing leads, so this table supports the agent role functionality.
